# Инструкции по работе с Git
## Команды
* git init — инициализация локального репозитория
* git status — получение информации о текущем состоянии репозитория
* git add — добавление файлов к следующему коммиту
* git commit -m "message" — создание коммита
* git log — вывод истории коммитов с их хеш-кодами. 
* git push - отправляет локальные изменения в удаленный репозиторий

### git init

Эта команда создаёт пустой репозиторий Git, то есть по существу только каталог .git с подкаталогами objects, refs/heads, refs/tags, и файлами шаблонов. Будет создана начальная ветка без коммитов (см. параметр --initial-branch для изменения её имени).
Если переменная среды GIT_DIR установлена, то она задаёт путь, который будет использоваться вместо .git в качестве базового пути для создания репозитория.
Если каталог хранения объектов задан с помощью переменной среды GIT_OBJECT_DIRECTORY, то каталоги с хеш-кодом sha1 будут создаваться в нём. В противном случае будет использоваться каталог по умолчанию, $GIT_DIR/objects.
Запускать git init внутри существующего репозитория безопасно: эта команда не будет перезаписывать то, что там уже есть. Основная цель, для которой это может потребоваться, состоит в том, чтобы подхватились новые шаблоны, добавленные с момента предыдущего запуска (или ещё, например, чтобы переместить репозиторий в другое место с помощью параметра --separate-git-dir).

### git status

в системе контроля версий Git показывает текущее состояние рабочего каталога и индекса (staging area). Она отображает:

* какие изменения были добавлены в индекс и готовы для коммита;
* какие изменения находятся в рабочем каталоге, но ещё не добавлены в индекс;
* какие файлы не отслеживаются Git (обычно новые файлы, добавленные в проект, или файлы, явно игнорируемые Git, указанные в файле .gitignore).

Команда не изменяет коммиты или изменения в локальном репозитории. 

### git add

в системе Git добавляет изменения из рабочего каталога в промежуточную зону (staging area) перед созданием коммита. Это позволяет:
* Выбрать, какие изменения войдут в следующий коммит. Это позволяет делать атомарные, логически завершённые коммиты, даже если внесено множество разных изменений в разные файлы.
* Перенести файлы в индекс — промежуточную зону перед репозиторием. После попадания в индекс файлы становятся подготовленными к коммиту (staged).

### git commit -m "message"

команда в Git, которая фиксирует (сохраняет) моментальный снимок (snapshot) текущего состояния ваших файлов в репозитории, создавая новую точку в истории проекта. Коммит содержит не только изменения кода, но и информацию об авторе, времени создания и, самое главное, осмысленное сообщение-описание того, что было изменено. Это позволяет отслеживать историю, откатываться к предыдущим версиям, понимать, кто и когда внес правки, и эффективно работать в команде. 

### git log

команда Git для просмотра истории коммитов в репозитории, отображающая последовательность изменений с информацией о каждом коммите (хэш, автор, дата, сообщение), начиная с самого свежего; она имеет множество параметров для форматирования, фильтрации и визуализации истории, например, --oneline для краткого обзора или --graph для дерева веток. 

### git push

команда Git, которая отправляет ваши локальные коммиты (сохраненные изменения) из вашего компьютера в удаленный репозиторий (например, на GitHub, GitLab, чтобы поделиться ими с командой или сохранить в облаке), синхронизируя их с общей версией проекта. Если вы работаете в команде, push делает вашу работу видимой для других; если в одиночку — обеспечивает резервное копирование кода.
